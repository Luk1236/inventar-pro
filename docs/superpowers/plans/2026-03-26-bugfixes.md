## Task 1: Frontend Login-URL-Fix

**Problem:** `app.config.js` fällt auf `http://192.168.1.100:8000` zurück wenn `EXPO_PUBLIC_BACKEND_URL` nicht gesetzt ist. `frontend/.env` setzt den Wert auf `http://localhost:8000`, aber Expo lädt `.env`-Dateien manchmal nicht zuverlässig beim Starten. Der Konflikt verhindert Login.

**Files:**
- Modify: `frontend/app.config.js`
- Verify: `frontend/.env`

- [ ] **Step 1: app.config.js reparieren**

Datei `frontend/app.config.js` — Zeile 17 ändern:

```js
// app.config.js – liest EXPO_PUBLIC_BACKEND_URL aus .env
//
// Lokal (Expo Go):       EXPO_PUBLIC_BACKEND_URL=http://192.168.1.xxx:8000
// Internet (Tunnel):     EXPO_PUBLIC_BACKEND_URL=https://equiptrack.example.com
// Web via Nginx:         EXPO_PUBLIC_BACKEND_URL=   (leer → relative URLs)

const appJson = require('./app.json');

const backendUrl = process.env.EXPO_PUBLIC_BACKEND_URL;

module.exports = {
  expo: {
    ...appJson.expo,
    extra: {
      // undefined → Fallback localhost; '' → relative URLs (Nginx-Betrieb)
      EXPO_PUBLIC_BACKEND_URL:
        backendUrl !== undefined ? backendUrl : 'http://localhost:8000',
    },
  },
};
```

- [ ] **Step 2: .env prüfen**

`frontend/.env` muss enthalten:
```
EXPO_PUBLIC_BACKEND_URL=http://localhost:8000
```

Falls du von einem echten Gerät oder Expo Go aus testest, statt `localhost` die IP des Rechners verwenden (z.B. `http://192.168.1.50:8000`). Die IP findest du mit:
- Windows: `ipconfig` → IPv4-Adresse
- Linux/Mac: `ip addr` oder `ifconfig`

- [ ] **Step 3: Expo-Cache leeren und neu starten**

```bash
cd frontend
npx expo start --clear
```

Erwartung: Beim Öffnen der App → Login mit `Admin` / `YNwJT56G` → Dashboard erscheint.

- [ ] **Step 4: Commit**

```bash
git add frontend/app.config.js
git commit -m "fix: correct fallback backend URL in app.config.js (was hardcoded Pi IP)"
```

## Task 2: CORS für Geräte im Netzwerk erweitern

**Problem:** `backend/.env` erlaubt nur `localhost`-Origins. Beim Zugriff von Handy, Tablet oder anderem Rechner im Netz schlägt CORS fehl.

**Files:**
- Modify: `backend/.env`

- [ ] **Step 1: ALLOWED_ORIGINS in backend/.env erweitern**

```env
# CORS: Komma-getrennte Liste erlaubter Origins (kein * verwenden!)
# Lokal + Expo Go + Raspi + LAN-Geräte:
ALLOWED_ORIGINS=http://localhost:8000,http://localhost:8081,http://localhost:19000,http://localhost:19006,http://192.168.1.100:8000,http://192.168.1.100:8081
```

> Hinweis: Wenn dein Rechner eine andere IP hat (z.B. `192.168.0.x`), passe die Origins entsprechend an. Die IP des Raspberry Pi kannst du mit `hostname -I` auf dem Pi ermitteln.

- [ ] **Step 2: Backend neu starten**

```bash
cd backend
python server.py
```

Erwartung: Keine CORS-Fehler mehr bei Zugriff vom Handy oder anderen Geräten im Netzwerk.

- [ ] **Step 3: Commit**

```bash
git add backend/.env
git commit -m "fix: extend CORS origins to allow LAN device access"
```

## Task 3: Access-Token-Dauer erhöhen

**Problem:** `ACCESS_TOKEN_EXPIRE_MINUTES=15` ist sehr kurz. Die Session-Warnung erscheint nach 13 Minuten. Im produktiven Betrieb ist das unpraktisch.

**Files:**
- Modify: `backend/.env`

- [ ] **Step 1: Token-Dauer in backend/.env anpassen**

```env
# Token Expiry
ACCESS_TOKEN_EXPIRE_MINUTES=60
REFRESH_TOKEN_EXPIRE_DAYS=30
```

- 60 Minuten Access-Token: Benutzer werden nicht nach kurzer Inaktivität ausgeloggt
- 30 Tage Refresh-Token: "Eingeloggt bleiben" funktioniert praktisch dauerhaft

- [ ] **Step 2: Backend neu starten**

```bash
cd backend
python server.py
```

Erwartung: Login bleibt 60 Minuten aktiv ohne erneute Anmeldung.

- [ ] **Step 3: Commit**

```bash
git add backend/.env
git commit -m "fix: increase access token expiry from 15min to 60min"
```

## Task 4: E-Mail-Benachrichtigungen einrichten

**Problem:** `send_maintenance_notification()` ist in `server.py` implementiert, aber SMTP-Credentials in `backend/.env` sind leer. `aiosmtplib` muss in `requirements.txt` vorhanden sein.

**Files:**
- Modify: `backend/.env`
- Verify: `backend/requirements.txt`

- [ ] **Step 1: aiosmtplib in requirements.txt prüfen**

```bash
grep -i aiosmtplib backend/requirements.txt
```

Falls nicht vorhanden, ergänzen:
```bash
echo "aiosmtplib>=3.0.0" >> backend/requirements.txt
pip install aiosmtplib
```

- [ ] **Step 2: SMTP-Credentials in backend/.env eintragen**

Für **Gmail** (empfohlen mit App-Passwort):

```env
# Email
ADMIN_EMAIL=deine-email@gmail.com
SMTP_SERVER=smtp.gmail.com
SMTP_PORT=587
SMTP_USERNAME=deine-email@gmail.com
SMTP_PASSWORD=xxxx-xxxx-xxxx-xxxx
NOTIFICATION_EMAIL=Lukas.Bootz@gmx.de
```

> **Gmail App-Passwort erstellen:**
> 1. Google-Konto → Sicherheit → 2-Schritt-Verifizierung aktivieren
> 2. Google-Konto → Sicherheit → App-Passwörter → Neues App-Passwort ("Mail", "Windows-Computer")
> 3. 16-stelligen Code kopieren → als `SMTP_PASSWORD` eintragen

Für **GMX** (da NOTIFICATION_EMAIL eine GMX-Adresse ist):
```env
SMTP_SERVER=mail.gmx.net
SMTP_PORT=587
SMTP_USERNAME=Lukas.Bootz@gmx.de
SMTP_PASSWORD=dein-gmx-passwort
```

- [ ] **Step 3: E-Mail manuell testen**

Backend starten und Wartungsalert auslösen. Oder direkt via Python testen:

```bash
cd backend
python -c "
import asyncio
from server import send_maintenance_notification
from datetime import datetime, timedelta
asyncio.run(send_maintenance_notification('Test-Gerät', datetime.now() + timedelta(days=3), is_overdue=False))
print('Test gesendet – prüfe deine E-Mail')
"
```

Erwartung: E-Mail kommt in der Inbox an. Falls nicht: SMTP-Credentials prüfen, Port 587 öffnen.

- [ ] **Step 4: Backend neu starten**

```bash
cd backend
python server.py
```

- [ ] **Step 5: Commit**

```bash
git add backend/requirements.txt backend/.env
git commit -m "feat: configure SMTP email notifications for maintenance alerts"
```
